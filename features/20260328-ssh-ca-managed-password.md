# Mars SSH CA Managed Password

## Summary

Change SSH CA creation so Mars generates a strong random password for the CA
private key instead of prompting the operator for one.

Mars stores that password encrypted in the backend next to the SSH CA files.
The password is encrypted with the environment data key, so it stays isolated
to the environment just like the rest of the Mars secret material.

This removes the need for the operator to manage the SSH CA password manually
while keeping the CA private key strongly encrypted at rest.

## Goals

- Auto-generate a strong password when creating a new SSH CA.
- Keep the SSH CA private key passphrase-protected.
- Store the generated password encrypted in the backend.
- Keep SSH CA password storage environment-scoped.
- Reuse the existing backend and secrets architecture.
- Prepare Mars for future in-memory SSH CA usage without persisting the
  password locally.

## Non-Goals

- Rotating or changing passwords for existing SSH CAs.
- Migrating older SSH CAs that were created with operator-provided passwords.
- Backfilling encrypted password objects for existing SSH CAs.
- Issuing SSH certificates in this feature.
- Persisting decrypted CA passwords locally.
- Exposing the decrypted password through CLI output.

## Current Problem

Today `mars ssh ca create` prompts the operator for the SSH CA password.

That creates a few problems:

- operators must manage the password outside Mars
- automation is harder because the password is not stored by Mars
- future CA operations must rely on manual password handling

Now that Mars already has:

- environment-scoped data keys
- backend storage abstraction
- binary-first encrypt and decrypt support

Mars can generate and store the password safely itself.

## Behavior Change

When the user runs:

```text
mars ssh ca create [name]
```

Mars must:

1. resolve the target environment
2. resolve the SSH CA name
3. check whether the SSH CA already exists locally or in backend
4. generate a strong random password
5. create the SSH CA private key using that password
6. encrypt the password with `SecretsService`
7. store the encrypted password object in the backend
8. upload the SSH CA private key and public key to the backend

The operator is no longer prompted for a password.

## Storage Layout

SSH CA durable backend storage now includes a third object:

- private key:
  - `mars/env/{env}/ssh/ca/{name}_ca_ed25519.key`
- public key:
  - `mars/env/{env}/ssh/ca/{name}_ca_ed25519.pub`
- encrypted password:
  - `mars/env/{env}/ssh/ca/{name}_ca_password.enc`

The password object is durable backend state and is part of the SSH CA record.

Local work-path cache remains unchanged in this feature:

- `<work_path>/env/{env}/ssh/ca/{name}_ca_ed25519.key`
- `<work_path>/env/{env}/ssh/ca/{name}_ca_ed25519.pub`

Mars does not persist the decrypted password to `work_path`.

## Password Object

The encrypted password file stores the encrypted secret record returned by
`SecretsService.encryptBytes(...)`.

That means the file should contain the full encrypted secret metadata Mars
needs later, not just ciphertext bytes.

Recommended shape:

```json
{
  "algorithm": "AES-GCM",
  "ciphertext": "base64...",
  "iv": "base64..."
}
```

Mars should serialize that encrypted secret record as JSON bytes and store it
at:

- `mars/env/{env}/ssh/ca/{name}_ca_password.enc`

## Password Generation

Mars generates the SSH CA password internally.

Requirements:

- use a cryptographically secure random source
- produce a password long enough for strong SSH private-key protection
- the password must be safe to pass to `ssh-keygen`
- the password is treated as opaque binary or text generated by Mars

Recommended V1 behavior:

- generate 32 random bytes
- encode them as base64url or hex for `ssh-keygen` compatibility

Exact encoding is implementation detail as long as it is:

- random
- strong
- deterministic only per generated value
- accepted by `ssh-keygen`

## Command Changes

### `mars ssh ca create [name]`

Behavior changes:

- if `name` is omitted, use `default`
- do not prompt the operator for a password
- generate the password automatically
- create the keypair with that generated password
- encrypt the password through `SecretsService`
- upload the encrypted password object to the backend

Existing duplicate checks still apply:

- if the local SSH CA files already exist in `work_path`, abort as already
  existing
- if the SSH CA already exists in backend, abort as already existing

The command should consider the SSH CA record complete only when all three
backend objects exist:

- private key
- public key
- encrypted password

## Backend Semantics

The encrypted password must go through the active `BackendService`.

That means:

- local backend stores it under `<work_path>/local/...`
- s3 backend stores it in the environment-scoped backend object path

`SshCaService` should not special-case S3 for this feature.
It should use the backend abstraction the same way the KV and secrets systems
do.

## Secrets Semantics

The generated password must be encrypted using the active `SecretsService`.

That means:

- the environment data key protects the SSH CA password
- backend only stores encrypted password material
- Mars can later decrypt the password in memory when CA operations need it

This feature depends on the current binary-first secrets API and uses it as the
standard path for SSH CA password storage.

## Existing SSH CAs

Existing SSH CAs created before this feature may not have:

- `mars/env/{env}/ssh/ca/{name}_ca_password.enc`

That is acceptable in this feature.

Rules:

- do not migrate them automatically
- do not rotate them automatically
- only newly created SSH CAs are guaranteed to have the encrypted password
  object

Future features can decide how to handle older SSH CAs that do not yet have a
managed password object.

## Service And Code Structure

Keep the logic under:

- `src/cli/app/ssh-ca/SshCaService`

Suggested responsibilities:

- generate SSH CA password
- create backend password object paths
- encrypt generated password via `SecretsService`
- store encrypted password bytes through `BackendService`
- include password-object existence in SSH CA create checks

Do not introduce a new standalone SSH password service in this feature.

## Suggested Implementation Shape

- add SSH CA password path helpers next to the existing SSH CA path helpers
- update `SshCaService.create(...)` to generate a password automatically
- remove the password prompt from the create command flow
- encrypt the password with `SecretsService.encryptBytes(...)`
- serialize the returned encrypted secret record to JSON bytes
- write the JSON bytes to backend as `{name}_ca_password.enc`
- update existence checks so the backend SSH CA record includes the password
  object
- update tests for create, show, pull, and destroy as needed

## Future Follow-Up

This feature sets up future work where Mars can:

- decrypt the managed SSH CA password in memory
- load the CA private key without operator-managed password state
- issue SSH certificates automatically

That future work should continue to avoid writing the decrypted password to
disk.

## Acceptance Criteria

- `mars ssh ca create [name]` no longer prompts for a password
- `mars ssh ca create [name]` generates a strong password automatically
- `mars ssh ca create [name]` uses the generated password with `ssh-keygen`
- `mars ssh ca create [name]` encrypts the password through `SecretsService`
- `mars ssh ca create [name]` stores the encrypted password object at
  `mars/env/{env}/ssh/ca/{name}_ca_password.enc`
- the encrypted password object is stored through the active `BackendService`
- newly created SSH CAs consist of:
  - private key
  - public key
  - encrypted password object
- the password is never written to local disk in decrypted form
- existing SSH CAs without encrypted password objects are left untouched
