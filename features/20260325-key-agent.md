# Mars Key Agent

## Summary

Add a background Mars key agent that keeps environment data keys in memory and
serves encrypt/decrypt requests over a local IPC channel.

The key agent uses the secrets-provider abstraction to load environment data
keys on demand and exposes a small JSON newline-delimited RPC interface.

This prevents the data key from touching disk or living in command-line
environment variables while still allowing Mars automation features to perform
crypto operations on demand.

## Goals

- Keep environment data keys in memory only.
- Avoid writing data keys to disk.
- Avoid putting data keys in environment variables.
- Reuse the same CLI binary for both normal command mode and key-agent mode.
- Automatically start the agent on demand when the CLI needs crypto access.
- Support project-scoped agent isolation using state tracked under `.mars`.
- Support agent lifecycle inspection and control via CLI commands.

## Non-Goals

- Supporting multiple different key agents per environment at the same time.
- Supporting remote key-agent communication.
- Supporting long-lived secrets outside Mars project scope.
- Implementing secret provider logic in this feature.
- Implementing backend persistence logic in this feature.

## Commands

Add a new top-level command group:

```text
mars key-agent
```

Subcommands:

- `mars key-agent ping`
- `mars key-agent show`
- `mars key-agent start`
- `mars key-agent stop`
- `mars key-agent serve`

## State Model

Do not use a separate `key-agent.json`.

Instead, extend `state.json` with a `key_agent` object.

Suggested shape:

```json
{
  "selected_environment": "infra/envs/dev/environment.yml",
  "key_agent": {
    "pid": 12345,
    "socket": "/tmp/mars-key-agent-abc123.sock",
    "token": "random-token"
  }
}
```

When the agent is not running:

```json
{
  "selected_environment": "infra/envs/dev/environment.yml",
  "key_agent": null
}
```

## Agent Identity

The key agent state contains:

- `socket: string`
- `pid: number`
- `token: string`

### Socket Paths

Use a random id per started agent instance:

- unix:
  - `/tmp/mars-key-agent-{random_id}.sock`
- windows:
  - `\\.\pipe\mars-key-agent-{random_id}`

### Token

The agent generates a random token and requires the client to send it with each
request.

This ensures only a process that can read the repo-local Mars state can talk to
the agent.

## Agent Lifecycle

### `mars key-agent start`

Start the key agent if one is not already running.

Behavior:

1. read Mars state
2. if `key_agent` exists, check whether the stored pid is still alive
3. if the pid is alive, do not start a second agent and print:
   - `key-agent running with {pid}`
4. if the pid is not alive, clear stale `key_agent` state
5. start the agent in the background by invoking:
   - `mars key-agent serve`
6. wait briefly for the agent to become reachable
7. do not write `key_agent` state directly from `start`
8. if ping succeeds within the timeout, print:
   - `key-agent started with {pid}`
9. if ping does not succeed within the timeout, print:
   - `failed to start key-agent. ping timeout`

### `mars key-agent serve`

Run the agent process itself.

Behavior:

1. if a valid agent already exists, exit
2. generate the socket path and token
3. start listening on the socket or pipe
4. once the socket is successfully bound, update `state.json` with `key_agent`
5. serve JSON-RPC-style newline-delimited requests
6. if the agent exits normally, clear `key_agent` back to `null` only when the
   stored `pid` and `token` still match the current agent process
7. before starting the server work, reconfigure LogTape with `reset: true` so
   the agent has its own logging setup
8. keep the console sink enabled for foreground troubleshooting runs
9. add a rotating file sink at `<work_path>/key-agent.log`
10. configure the rotating file sink with:
    - max size `1 MiB`
    - max files `3`
11. use the verbose formatter for the key-agent serve logging configuration so
    foreground troubleshooting output is log-like and detailed
12. the key-agent server should log:
    - server start
    - server shutdown
    - each received request
    - request failures
    - idle-timeout shutdown

This command is used internally by Mars and may also be invoked directly for
debugging.

### `mars key-agent stop`

Stop the running key agent for the current project.

Behavior:

1. read state
2. if `key_agent` is `null`, print:
   - `key-agent not running`
3. if pid is alive, first send an authenticated `shutdown` RPC request
4. wait up to 5 seconds
5. if the agent is still running after the timeout, force kill it by pid
6. print:
   - `stopping key-agent {pid}`
7. the agent is responsible for clearing `key_agent` in state when it exits

### `mars key-agent show`

Show whether the agent is currently running and which pid it is using.

If running, print:

- `pid: {pid}`
- `socket: {socket}`

If stopped, print:

- `key-agent not running`

### `mars key-agent ping`

Ping the currently running key-agent.

If the agent is running and responds, print:

- `ping sucessfully`

If the agent is running but the ping fails, print:

- `ping failed: {error}`

If the agent is not running, print:

- `key-agent is not running`

## Idle Shutdown

If the key agent is idle for more than 5 minutes, it must shut itself down and
clear the `key_agent` state.

Idle means:

- no requests received during that time window

## Client Behavior

When normal Mars CLI commands need crypto operations:

1. read state
2. check whether `key_agent` exists
3. verify the stored pid is alive
4. verify the socket responds to a `ping` request
5. if the socket does not respond, assume the agent is dead, clear state, stop
   the stale process if needed, and start a new one
6. if the agent is missing or invalid, start it
7. retry up to 5 times with increasing delays
7. once the agent responds, send encrypt/decrypt requests as needed

## IPC Protocol

Use a simple JSON newline-delimited RPC protocol.

Each request is one JSON object terminated by a newline.

Each response is one JSON object terminated by a newline.

### Transport Envelope

The generic JSON-RPC transport should wrap messages in an envelope:

```json
{
  "id": 1,
  "message": {
    "type": "ping",
    "token": "random-token"
  }
}
```

The `id` is transport-level request tracking.

The key-agent request and response payloads live inside the `message` field.

### Common Request Fields

- `type`
- `token`
- `environment`

### Request Types

- `ping`
- `encrypt`
- `decrypt`
- `shutdown`

### Encrypt Request

Suggested shape:

```json
{
  "type": "encrypt",
  "token": "random-token",
  "environment": "gl-dev",
  "plaintext": "base64..."
}
```

`plaintext` is base64-encoded.

### Encrypt Response

Suggested shape:

```json
{
  "ok": true,
  "type": "encrypt",
  "encrypted_secret": {
    "algorithm": "AES-GCM",
    "ciphertext": "base64...",
    "iv": "base64..."
  }
}
```

### Decrypt Request

Suggested shape:

```json
{
  "type": "decrypt",
  "token": "random-token",
  "environment": "gl-dev",
  "encrypted_secret": {
    "algorithm": "AES-GCM",
    "ciphertext": "base64...",
    "iv": "base64..."
  }
}
```

### Decrypt Response

Suggested shape:

```json
{
  "ok": true,
  "type": "decrypt",
  "plaintext": "base64..."
}
```

### Error Response Shape

The only shared response shape should be the error response.

Suggested shape:

```json
{
  "ok": false,
  "type": "encrypt",
  "error": "message"
}
```

## Data Key Behavior

The key agent loads environment data keys on demand.

Requests must include the target environment id so the agent knows which
environment data key to load or reuse.

The data key should be cached in memory per environment while the agent is
running.

The secrets-provider abstraction is responsible for obtaining access to that
environment's data key.

The key agent is responsible only for:

- requesting the data key from the provider
- holding it in memory
- using it for encrypt/decrypt operations

## State Update Rules

`StateService` should not overwrite the entire state file when updating one
field.

State updates must:

- read the existing Mars state
- merge the new state field into the existing object
- write the merged state back

This is required because multiple Mars features may store state in the same
`state.json`, including:

- `selected_environment`
- `key_agent`

## Service And Code Structure

### Key Agent App Area

Use:

- `src/app/key-agent/key-agent-service.ts`
- `src/app/key-agent/key-agent-manager.ts`
- `src/app/key-agent/key-agent-server.ts`
- `src/app/key-agent/key-agent-client.ts`

### Generic Transport And Coordination Helpers

Use:

- `src/lib/json-rpc-shapes.ts`
- `src/lib/json-rpc-server.ts`
- `src/lib/json-rpc-client.ts`
- `src/lib/promise-signal.ts`

### Responsibilities

#### `key-agent-service.ts`

- load data keys on demand
- cache data keys in memory by environment

#### `key-agent-server.ts`

- own key-agent-specific server orchestration
- enforce idle shutdown
- remove stale socket files
- coordinate key-agent state registration and cleanup
- write runtime server logs

The key-agent server should use a generic `JsonRpcServer` internally rather
than implementing the socket framing and connection bookkeeping itself.

#### `key-agent-client.ts`

- adapt key-agent request and response shapes onto the generic `JsonRpcClient`
- materialize typed response payloads for callers

#### `key-agent-manager.ts`

- start the background process
- stop the background process
- inspect whether the stored pid is alive
- verify the socket is responsive
- retry startup and ping flow
- coordinate state updates related to `key_agent`

#### `json-rpc-server.ts`

- own the underlying socket listener
- track active sockets by internal numeric `socket_id`
- keep a per-socket data buffer
- frame newline-delimited messages
- parse envelopes
- emit parsed message events with:
  - `socket_id`
  - `envelope_id`
  - `message`
- expose `respond(socketId, envelopeId, message)` so callers do not touch raw
  sockets directly
- close and clean up all active sockets when shutting down

#### `json-rpc-client.ts`

- create the socket connection on demand
- keep the socket connection open and reuse it across requests
- assign envelope ids internally for each request
- track pending requests by envelope id
- route responses back to the right pending request
- timeout requests after 5 seconds
- close the transport connection after 30 seconds of inactivity

#### `promise-signal.ts`

- provide a reusable wait/resolve/reject primitive for coordination points like
  server shutdown

### Secrets Integration

`SecretsService` should remain an interface.

`KeyAgentSecretsService` should be an implementation of that interface.

The client-facing secrets service should use the key-agent manager to ensure an
agent is available, then send RPC requests through the socket.

`KeyAgentSecretsService` should cache the `KeyAgentClient` by socket path when
possible so repeated operations can reuse the same transport connection.

The key-agent implementation should resolve the active secrets provider through
`SecretsProviderFactory`, so the running agent uses the configured provider
without hardcoding password or KMS branching in the agent logic.

## Suggested Implementation Shape

- extend `state.json` with `key_agent`
- add the key-agent command tree
- add the key-agent manager
- add the key-agent client
- add the key-agent server
- add the key-agent service
- add the generic JSON-RPC client/server transport helpers
- add the promise signal helper
- update `StateService` so state writes merge instead of replacing the whole
  file
- implement socket or pipe creation
- implement token generation and validation
- implement `ping`, `encrypt`, `decrypt`, and `shutdown` requests
- implement pid liveness checks
- implement agent auto-start with retry and backoff
- implement 5-minute idle shutdown

## Crypto Primitive

Once the agent has the environment data key, use a standard authenticated
encryption mode such as:

- `AES-256-GCM`

Prefer built-in Bun or Web Crypto primitives where possible instead of adding
an external crypto package.

## Acceptance Criteria

- Mars stores key-agent state inside `state.json` under `key_agent`
- `mars key-agent start` does not start a second agent when a valid one already
  exists
- `mars key-agent start` prints `key-agent running with {pid}` when already
  running
- `mars key-agent start` prints `key-agent started with {pid}` when it starts a
  new agent
- `mars key-agent start` prints `failed to start key-agent. ping timeout` when
  the agent does not answer `ping` in time
- `mars key-agent start` clears stale agent state before replacing it
- `mars key-agent serve` registers itself in state with `socket`, `pid`, and
  `token`
- `mars key-agent serve` writes `key_agent` only after the socket is bound and
  ready
- `mars key-agent serve` only clears `key_agent` on exit when the stored `pid`
  and `token` still match the current process
- `mars key-agent show` reports whether the agent is running and which pid it
  uses
- `mars key-agent show` prints `key-agent not running` when stopped
- `mars key-agent ping` prints `ping sucessfully` when the running agent
  responds
- `mars key-agent ping` prints `ping failed: {error}` when the running agent
  does not respond successfully
- `mars key-agent ping` prints `key-agent is not running` when there is no
  agent
- `mars key-agent stop` stops the running agent and clears `key_agent`
- `mars key-agent stop` first sends `shutdown`, then waits up to 5 seconds, and
  force kills the agent if needed
- `mars key-agent stop` prints `stopping key-agent {pid}` when stopping a
  running agent
- `mars key-agent stop` prints `key-agent not running` when there is no agent
- the client can verify the agent by checking both pid liveness and socket
  `ping`
- when the pid exists but the socket does not respond, Mars treats the agent as
  stale and starts a new one
- the client retries startup and ping up to 5 times with increasing delays
- the agent accepts newline-delimited JSON requests
- transport messages are wrapped in envelopes with an internal `id`
- the agent requires the correct token on each request
- the agent supports `ping`, `encrypt`, `decrypt`, and `shutdown`
- the JSON-RPC server emits parsed message events with `socket_id`,
  `envelope_id`, and `message`
- the JSON-RPC server exposes `respond(socketId, envelopeId, message)` instead
  of exposing raw sockets to subscribers
- the JSON-RPC client reuses a live socket connection instead of creating a new
  one for every request
- the JSON-RPC client closes its connection after 30 seconds of inactivity
- `mars key-agent serve` resets LogTape configuration before the server starts
- `mars key-agent serve` writes logs to `<work_path>/key-agent.log`
- the key-agent log uses a rotating file sink with `1 MiB` max size and `3`
  files retained
- the key-agent serve logging keeps console output enabled for foreground
  troubleshooting
- the key-agent server logs start, shutdown, received requests, request
  failures, and idle-timeout shutdown
- `encrypt` returns a full encrypted-secret record including algorithm, iv, and
  ciphertext
- `decrypt` accepts a full encrypted-secret record including algorithm, iv, and
  ciphertext
- requests include the target environment so the agent can load the right data
  key
- environment data keys remain in memory only while the agent is running
- the agent shuts down after 5 minutes of inactivity
- `StateService` merges state updates instead of overwriting unrelated state
  fields
- encryption uses a standard authenticated mode such as `AES-256-GCM`
