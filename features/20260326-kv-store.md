# KV Store

## Summary

Add a Mars-managed per-environment key-value store so operators and automation
can place configuration, secrets, URLs, certificates, and file-backed values
into one consistent system.

The KV store builds on the backend and secrets abstractions we already have:

- `BackendService` determines where durable KV data lives
- `SecretsService` / secrets providers determine how sensitive values are
  encrypted
- each environment has its own isolated KV store
- Mars pulls KV state into the local work path, operates on it locally, and
  syncs it back to the backend

This document reflects the current V1 design direction for implementation.

## Goals

- Provide a unified store for environment-scoped values
- Support both secret and non-secret values in one system
- Keep data portable across local and centralized backends
- Allow Mars to later project values into deployment targets such as:
  - cloud-init
  - SSM
  - Secrets Manager
  - Docker secrets
  - runtime config files
- Make KV usable both by operators and automation

## Non-Goals

- This feature does not yet define the projection/deployment side
- This feature does not yet define GitHub Actions integration in detail
- This feature does not yet define multi-environment transactions
- This feature does not yet define garbage collection or retention cleanup
- This feature does not yet define cross-backend migration workflows

## Key Ideas

### Environment Isolation

Each environment has its own independent KV store.

That means:

- `gl-dev` KV data is isolated from `gl-test`
- secrets and config for one environment never share the same SQLite file
- local cache and backend layout stay aligned with the rest of Mars

### Backend-Agnostic Durable Storage

KV durable state goes through the active backend implementation.

- local backend stores under `<work_path>/local/...`
- s3 backend stores under the environment-scoped S3 layout

The KV feature should not talk to S3 or local files directly for durable
storage. It should go through backend abstractions.

### Local Working Copy

Mars operates on a pulled local working copy under:

```text
<work_path>/env/<env>/kv/
```

Suggested local layout:

```text
<work_path>/env/<env>/kv/store.db
<work_path>/env/<env>/kv/store.db-wal
<work_path>/env/<env>/kv/store.db-shm
<work_path>/env/<env>/kv/blobs/<uuid>
```

### SQLite As Source Of Truth

SQLite is the source of truth for KV metadata and lookup state.

V1 model:

- SQLite stores keys, versions, metadata, and blob references
- large payloads live outside SQLite as blob files
- SQLite references those blobs by `data_blob_id`
- Bun's built-in SQLite driver is used for the database layer
- all SQLite access goes through a small injected `DbClient` wrapper

## Key Path Model

Keys are hierarchical paths:

```text
/service/component/resource
```

Rules:

- case-sensitive
- normalized
- must start with `/`
- no trailing `/`
- no empty segments
- path segments may contain only letters, numbers, `_`, and `-`
- no `#`
- no `@`
- path is the logical key identity

Examples:

```text
/myservice/db/password
/myservice/api/url
/myservice/tls/cert
```

## Recommended V1 Scope

V1 keeps versions and defers labels entirely.

That means:

- every `set` creates a new immutable version
- labels are out of scope for now
- `rotate` and `revert` are out of scope for now
- version GC is out of scope for now

Version retention behavior in V1 is simple:

- any write bumps `version_id`
- old versions remain
- no GC or retention pruning happens yet

This keeps the first version focused and avoids guessing at label semantics too
early.

## Proposed SQLite Model

### kv_key

```sql
CREATE TABLE kv_key (
  key_path TEXT PRIMARY KEY,
  create_date TEXT NOT NULL,
  update_date TEXT NOT NULL
);
```

### kv_key_version

```sql
CREATE TABLE kv_key_version (
  key_path TEXT NOT NULL,
  version_id INTEGER NOT NULL,
  secret_config TEXT,
  data_type TEXT NOT NULL,
  data_content BLOB,
  data_blob_id TEXT,
  data_size INTEGER NOT NULL,
  create_date TEXT NOT NULL,
  PRIMARY KEY (key_path, version_id)
);
```

Suggested `data_type` values:

- `text`
- `file`

### kv_pending_blob

```sql
CREATE TABLE kv_pending_blob (
  key_path TEXT NOT NULL,
  operation TEXT NOT NULL,
  local_path TEXT NOT NULL,
  PRIMARY KEY (key_path, operation, local_path)
);
```

Suggested `operation` values:

- `upload`
- `delete`

### migrations

SQLite schema should be managed through explicit migrations.

Suggested area:

```text
src/cli/app/kv/migrations/
```

The migration framework itself should live in:

```text
src/lib/migration/
```

Suggested base class:

```ts
export abstract class DbMigration {
  key: string;

  constructor(key: string) {
    this.key = key;
  }

  abstract apply(db: DbClient): void;
  abstract revert(db: DbClient): void;
}
```

Suggested migrator behavior:

- keeps a migration table with `key` and `apply_date`
- applies pending migrations inside a transaction
- can revert in reverse order

Migration files should use:

```text
<date>-<name>.ts
```

Example:

```text
20260301-init-kv.ts
```

## Blob Storage

Suggested durable backend-relative blob layout:

```text
mars/env/<env>/kv/blobs/<uuid>
```

For the local backend, the same logical paths should map under:

```text
<work_path>/local/mars/env/<env>/kv/...
```

Blob path is intentionally opaque. We do not use `.blob` or `.enc` suffixes in
V1. Any encryption metadata lives in `kv_key_version.secret_config`.

The local state blob path lives under:

```text
<work_path>/env/<env>/kv/blobs/<uuid>
```

Whenever a value is stored externally as a file/blob, Mars writes it to the
local state blob path first and records a `kv_pending_blob` entry so state save
can later upload or delete it in the backend.

## Encryption Model

V1 uses the environment data key directly.

That means:

- no per-key DEKs
- no per-version DEKs
- secret values are encrypted with the environment data key we already get from
  the secrets architecture
- `secret_config` stores the encryption metadata needed to decrypt later

Later we can add per-key or per-version DEKs and offer a re-encode migration
path if needed.

Secret metadata such as algorithm and IV is stored in
`kv_key_version.secret_config`.

## Backend Layout

Suggested backend-relative KV layout:

```text
mars/env/<env>/kv/store.db
mars/env/<env>/kv/store.db-wal
mars/env/<env>/kv/store.db-shm
mars/env/<env>/kv/blobs/<uuid>
```

This keeps KV aligned with the existing environment-scoped durable storage
layout.

## Sync Model

V1 uses staged local changes plus explicit state save.

### V1 Sync Flow

Flow:

1. pull `store.db` and needed blobs locally
2. mutate locally
3. leave changes staged in local cache
4. run `mars kv state save`

We explicitly do not use Litestream in V1.

`mars kv state pull` only refreshes the DB. Blobs remain lazy and are fetched
on demand.

Any KV command that needs the local DB state should automatically pull it first
when the local state is missing.

If local state already exists, Mars must not auto-pull and overwrite staged
local changes. `mars kv state pull` must also skip replacing the local DB when
`store.db` already exists in the work path. The operator must use
`mars kv state clear` to discard
local staged state before pulling fresh state again.

## Locking Model

Suggested backend-relative lock path:

```text
mars/env/<env>/lock/<name>.json
```

Suggested lock fields:

```json
{
  "holder": "hostname-or-process-id",
  "token": "random-token",
  "expire_at": "2026-03-26T12:00:00.000Z"
}
```

- local backend: no remote lock needed
- s3 backend: add a simple lease-style lock before state save writeback

We do need locking in V1 so multiple operators or automation jobs do not
clobber each other.

### Recommended S3 Lease Lock

Recommended V1 behavior:

1. lock object lives at `mars/env/<env>/lock/<name>.json`
2. Mars writes a random `token`, holder string, and `expire_at`
3. before taking a lock:
   - if lock does not exist, create it
   - if lock exists but `expire_at` is in the past, overwrite it
   - if lock exists and is not expired, fail acquisition
4. after writing lock content, Mars reads it back and verifies the token still
   matches
5. on release, Mars only deletes the lock if the stored token still matches the
   current holder

This is not perfect distributed locking, but it is likely good enough for V1
and matches the repo's current simplicity level.

Recommended V1 lease TTL:

- `3 minutes`
- no lease renewal in V1

## Commands

Initial command set:

```text
mars kv set <key_path>
mars kv get <key_path>
mars kv show <key_path>
mars kv list <key_path?>
mars kv remove <key_path>
mars kv state pull
mars kv state save
mars kv state clear
```

### `mars kv set`

Final V1 shape:

```text
mars kv set <key_path>
  --value <data>
  --file <path>
  --input
  --secret
```

Rules:

- exactly one of `--value`, `--file`, or `--input` must be provided
- `--secret` means the stored value is encrypted
- `--value` provides inline text data
- `--file` reads the value from a local file path
- `--input` reads the value from stdin
- `--file` implies the internal KV type is `file`
- `--value` and `--input` imply the internal KV type is `text`
- values `<= 16 KiB` are stored inline unless type is `file`
- values `> 16 KiB` are stored as external blobs
- `--secret` does not force blob storage by itself
- when stored as an external blob, the local blob file is written first and a
  `kv_pending_blob` upload entry is created

### `mars kv get`

```text
mars kv get <key_path>
  --raw
```

Rules:

- resolves a key path or explicit version
- returns plaintext for secrets because it is an explicit retrieval command
- `--raw` writes directly to stdout without adding a newline
- for binary file values, `--raw` writes the bytes as-is so callers can pipe to
  a file

### `mars kv show`

Shows metadata such as:

- key
- current version
- whether secret
- value kind
- size
- create_date
- update_date

`set`, `get`, `show`, and `remove` all operate on an explicit key path only.

### `mars kv list`

Lists child keys under a prefix.

Output columns:

- `key`
- `type`
- `size`
- `secret`
- `date`

Where:

- `type` is `text` or `file`
- `secret` is shown as `yes` or `no`
- `date` is the latest `update_date`

Examples:

```text
mars kv list /
mars kv list /myservice
```

`list` is the only V1 KV command that uses prefix semantics.

### `mars kv remove`

Removes the logical key.

- V1 uses physical delete
- no tombstones in V1
- if versions reference external blobs, removal creates `kv_pending_blob`
  delete entries for those local/backend blob paths

### `mars kv state pull`

- refreshes the local DB from the backend
- skips the pull when local `store.db` already exists
- does not eagerly pull blobs
- KV commands that need DB state should auto-pull as needed

### `mars kv state save`

Flow:

1. acquire backend lock when needed
2. checkpoint the local SQLite DB
3. upload pending `upload` blobs
4. delete pending `delete` blobs from backend
5. clear processed `kv_pending_blob` rows locally
6. checkpoint the local SQLite DB again
7. sync `store.db`
8. release backend lock

### `mars kv state clear`

- clears the local state DB and local blob cache
- does not affect durable backend state
- useful when the operator wants to discard local staged state and refresh later
- behaves like a local reset back to empty state and does not touch
  durable backend state

## Secrets API Change

The current secrets architecture is text-oriented, but KV needs:

- bytes encryption and decryption
- metadata records returned with encrypted blob payloads

We do not need bulk secret operations in V1.

KV pushes the secrets layer toward a binary-first API.

That means the secrets service and provider surface should move from a
text-oriented contract to a bytes-oriented contract for the core crypto path.

Recommended direction:

- binary encrypt input
- binary decrypt output
- return encryption metadata as a structured record
- let text encryption/decryption become a convenience wrapper on top of the
  binary API, not the core primitive

Why:

- KV stores both inline values and file/blob values
- file/blob values need byte-safe encryption and decryption
- secret metadata such as algorithm and IV already belongs in
  `kv_key_version.secret_config`
- a binary-first contract is a better fit for both KV and future secret/file
  use cases

V1 expectation:

- add binary encrypt/decrypt methods to the secrets service/provider stack
- keep text helpers as wrappers where still useful
- use the binary result plus metadata to populate:
  - `kv_key_version.data_content`
  - `kv_key_version.secret_config`
  - external blob contents when `data_type = file`

## Backend Notes

The backend abstraction remains file-oriented for KV.

For V1:

- KV does not need backend list-by-prefix support because listing comes from
  SQLite queries
- local backend does not need remote locking
- s3 backend needs lease lock acquire and release behavior for state save
- V1 lock acquisition uses the shared `LockService`
- `LockService` takes `(environment, lockName)` and stores lock files at
  `env/<env>/lock/<name>.json`
- `LockService.acquire(...)` returns a result instead of throwing so the caller
  can decide how to handle lock contention
- V1 lock acquisition continues to use read-check-write plus read-back token
  verification
