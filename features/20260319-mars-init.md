# Mars Init Base Config And Workspace Setup

## Summary

Add an initial `mars init` CLI command that prepares the minimum local project
setup Mars needs to operate safely inside a repository.

This command creates:

- `mars.config.json`
- `.mars/`

Mars works locally and may later push or pull state from a centralized source
such as S3, but this feature only covers the local bootstrap required to begin
working in a repo.

## Goals

- Give users a single command to initialize Mars in a repository.
- Create a predictable local config file with sensible defaults.
- Create a dedicated local workspace folder for Mars operations.
- Keep Mars work isolated from the rest of the repository.
- Make the command safe to run multiple times.

## Non-Goals

- Pushing or pulling state from S3 or any other remote backend.
- Validating that `stack_path` exists during init.
- Creating stack folders or any infrastructure content.
- Adding advanced config fields beyond the two initial properties.
- Implementing other CLI commands that consume the config.

## User Story

As a user setting up Mars in a repository, I want to run `mars init` once and
have Mars create its base config and local workspace so future commands can
work without modifying the rest of the repo directly.

## Command Behavior

### Command

The CLI exposes:

- `mars init`

### Successful Init

When `mars init` runs in a repository root:

1. If `mars.config.json` does not exist, Mars creates it.
2. Mars creates the local work directory defined by `work_path`.
3. Mars ensures the work directory exists even when it is using the default
   path.

### Re-Running Init

`mars init` must be idempotent.

When the command is run again:

- If `mars.config.json` already exists, Mars must not overwrite it.
- If the configured work directory already exists, Mars leaves it in place.
- The command should still complete successfully when the repo is already
  initialized.

## Config Contract

### File Name

- `mars.config.json`

### Initial Shape

The config contains these properties:

- `stack_path: string`
- `work_path: string`

### Defaults Written By Init

When Mars creates a new config file, it writes:

```json
{
  "stack_path": "infra/stacks",
  "work_path": ".mars"
}
```

### Property Rules

- `stack_path` is required configuration.
- `work_path` is optional at the product level and defaults to `.mars`.
- `mars init` writes `work_path` explicitly into the generated config so the
  initial file is concrete and easy to understand.

## Workspace Folder

### Purpose

The work folder is a Mars-owned workspace where the tool can perform local work
without affecting other files in the repository.

### Default Path

- `.mars`

### Requirements

- The folder must be created by `mars init`.
- The folder must be treated as local-only workspace state.
- The folder is intended to stay out of version control, but `mars init` does
  not modify `.gitignore`.

## Boot And App Structure

This setup belongs in the CLI boot layer.

Implementation should place the CLI-specific setup in `src/cli/boot/`, because
each app surface such as CLI, web, or service owns its own boot folder and
startup wiring.

The entrypoint should remain thin and delegate setup and command assembly into
CLI boot helpers.

## Suggested Implementation Shape

This section is guidance for implementation and may be adjusted during coding
if the codebase suggests a better naming detail while keeping behavior the
same.

- Add a CLI command for `mars init`.
- Add CLI boot helpers under `src/cli/boot/` for config and program setup.
- Add a config model or parser for `mars.config.json`.
- Add filesystem helpers responsible for:
  - creating `mars.config.json` when missing
  - creating the work directory when missing

## Acceptance Criteria

- Running `mars init` in a repo without Mars setup creates `mars.config.json`.
- The created config uses `stack_path: "infra/stacks"`.
- The created config uses `work_path: ".mars"`.
- Running `mars init` creates the `.mars/` directory.
- Running `mars init` does not overwrite an existing `mars.config.json`.
- Running `mars init` succeeds when `.mars/` already exists.
- Running `mars init` reports a simple per-item result such as `created
  mars.config.json` or `skipped .mars (already exists)`.
- The init setup logic is wired through `src/cli/boot/`.

## Output Behavior

The command should emit simple human-readable status lines describing what it
did for each item it manages.

Examples:

- `created mars.config.json`
- `skipped mars.config.json (already exists)`
- `created .mars`
- `skipped .mars (already exists)`
